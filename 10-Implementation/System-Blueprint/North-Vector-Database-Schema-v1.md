# North Vector Database Schema v1.0

## Status

**Superseded in part as of 2026-07-03.** The relational schema below was never implemented — PostgreSQL was deleted before real migrations existed. See `10-Implementation/ADRs/ADR-0101-Use-Firestore-as-the-Primary-Database.md`. A new section, "Actual Firestore Shape (2026-07-03)", documents what's really deployed for the six collections that exist. Entities that were never built (memories, plans, approvals, executions, reviews) are marked "Not Yet Designed for Firestore" rather than translated speculatively — inventing a NoSQL schema for something nobody has decided to build yet would just create a second stale document later. The original relational design below is left in place for historical context, not deleted.

## Purpose

This document translates the North Vector Domain Model and ERD into an initial relational database schema.

It defines the first Postgres-oriented structure for the core North Vector platform.

This is still a design document, not a migration file. **It was never implemented.**

# Actual Firestore Shape (2026-07-03)

Firestore has no schema enforcement, no foreign keys, and no junction tables in the relational sense. Each collection below is documented as it actually exists in the deployed store modules (`lib/task-store.ts`, `lib/goal-store.ts`, `lib/project-store.ts`, `lib/decision-memory.ts`) or Firestore-writing code (`app/api/v1/plaid/exchange/route.ts`, `functions/src/index.ts`), not translated from the SQL design below.

## Collection: `tasks`

Document ID: auto-generated by Firestore (`addDoc`). Written/read via `lib/task-store.ts`.

```text
{
  title: string
  description: string
  status: "scheduled" | "active" | "completed" | "paused" | "cancelled"
  priority: "low" | "medium" | "high" | "critical"
  energy: "low" | "medium" | "high"
  domain: "academic" | "career" | "health" | "personal" | "north-vector"
  goalId?: string        // no foreign key enforcement — a plain string field
  projectId?: string     // same
  dueDate?: string       // ISO date string, not a Firestore Timestamp
  estimatedMinutes?: number
  notes?: string
  createdAt: string      // ISO string, set client-side via new Date().toISOString()
  updatedAt: string
  completedAt?: string | null
}
```

Notes: `goalId`/`projectId` are unvalidated string references, not enforced relationships — a task can reference a goal ID that doesn't exist, and nothing would catch it. `createdAt`/`updatedAt` are client-set ISO strings, not Firestore's `serverTimestamp()` (contrast with `daily-runs` below, which does use `serverTimestamp()` deliberately).

## Collection: `goals`

Document ID: auto-generated. Written/read via `lib/goal-store.ts`.

```text
{
  title: string
  description: string
  horizon: "now" | "mid" | "long"
  status: "active" | "completed" | "paused" | "cancelled"
  progress: number       // 0-100, no enforcement of range
  targetDate?: string    // ISO date string
  keyResults: string[]
  risk: "low" | "medium" | "high"
  createdAt: string
  updatedAt: string
}
```

## Collection: `projects`

Document ID: auto-generated. Written/read via `lib/project-store.ts`.

```text
{
  title: string
  description: string
  status: "planning" | "active" | "completed" | "paused" | "cancelled"
  progress: number
  goalId?: string         // unvalidated reference, same caveat as tasks.goalId
  taskCount: number       // denormalized count, not derived from a query
  doneCount: number       // denormalized count
  nextMilestone?: string
  tags: string[]
  createdAt: string
  updatedAt: string
}
```

Notes: `taskCount`/`doneCount` are denormalized fields set at write time, not computed from the `tasks` collection — nothing currently keeps them in sync with actual task state.

## Collection: `decisions`

Document ID is **not** auto-generated — it's a normalized version of the question text (lowercased, punctuation stripped), used as a deliberate natural key so repeat questions hit the same document. Written/read via `lib/decision-memory.ts`.

```text
{
  question: string          // original, unnormalized question text
  recommendation: string
  reasons: string[]
  risks: string[]
  confidence: string
  createdAt: string
  timesAsked: number        // incremented via Firestore's increment() on repeat lookups
}
```

This is a genuinely different pattern from `tasks`/`goals`/`projects` — deterministic document IDs instead of auto-generated ones, chosen specifically to make "has this question been asked before" a direct document lookup instead of a query.

## Collection: `plaid_items`

Document ID: auto-generated. Written via `app/api/v1/plaid/exchange/route.ts` using the **Admin SDK**, not the client SDK — this collection is deliberately unreachable from any client-side code path (see `firestore.rules`: `allow read, write: if false` for this collection specifically, overriding the general owner-email rule).

```text
{
  itemId: string              // Plaid's item_id, not sensitive
  accessTokenEncrypted: string  // AES-256-GCM ciphertext — see lib/encryption.ts; never stored in plaintext
  ownerUid: string             // Firebase Auth UID of the caller at write time
  createdAt: string
}
```

## Collection: `daily-runs`

Document ID: auto-generated. Written by the `dailyRiskScan` Cloud Function (`functions/src/index.ts`) via the Admin SDK. Client writes are denied by `firestore.rules`; client **reads** are allowed for the owner (unlike `plaid_items`, which denies both).

```text
{
  timestamp: Timestamp     // Firestore serverTimestamp() — assigned by Firestore's backend at write-commit time, not the function instance's clock
  risks: RiskRecord[]      // output of lib/risk-engine.ts's evaluateRisks(), embedded directly, not referenced
  tasksEvaluated: number
  goalsEvaluated: number
}
```

Notes: `risks` is an array of full risk objects embedded in the document (denormalized), not a reference to separate risk documents — appropriate for Firestore, where subcollections/references would be unnecessary overhead for data that's always read together with its parent.

## Not Yet Designed for Firestore

`memories`, `plans`, `approvals`, `executions`, and `reviews` were never implemented against any database — not PostgreSQL (no real migrations were ever run) and not Firestore. The relational design in the sections below for these entities was speculative when written and remains speculative now. No Firestore document shape has been decided for them. Do not treat the SQL below as a blueprint to translate — if/when these are built, they deserve their own design pass informed by how they're actually used, the same way `tasks`/`goals`/`projects`/`decisions` were shaped around real application code rather than a pre-drawn schema.

## Relationships and Junction Tables in Firestore

The relational design below uses junction tables (`goal_projects`, `goal_tasks`, `goal_plans`) and polymorphic entity tables (`memory_entities`, `review_entities`) for many-to-many relationships. None of this exists in the actual Firestore implementation. Where a relationship exists at all today, it's a plain unvalidated string field (`goalId` on `tasks` and `projects`) — not a table, not a validated reference, not queryable in reverse without a collection scan. This is a real, accepted gap (see ADR-0101's "Negative Consequences" and "Residual Risks"), not an oversight to silently paper over here.

# Schema Principles

The database schema should support:

- clear ownership by user
- durable long-term memory
- explicit goal and project tracking
- planning and task execution
- approval-gated actions
- review and reflection cycles
- future auditability
- future event-driven architecture

The schema should remain simple enough for V1 while leaving room for future expansion.

# Core Tables

## users

**Not implemented as a Firestore collection.** There is exactly one user, identified by Firebase Auth (email/password) rather than an application-owned `users` table — see `10-Implementation/ADRs/ADR-0102-Use-Firebase-Auth-for-Identity.md`. The owner's identity (email, UID) is enforced via `firestore.rules` and a hardcoded owner-email constant (`lib/owner.ts`), not stored as a queryable Firestore document. `name`, `preferred_name`, `timezone`, and `status` as designed below have no Firestore equivalent — profile data of this kind was never built.

Represents the human being served by North Vector (as originally designed):

```sql
users (
  id uuid primary key,
  name text not null,
  preferred_name text,
  email text unique,
  timezone text,
  status text not null,
  created_at timestamptz not null,
  updated_at timestamptz not null
)
```

Recommended statuses:

```text
active
inactive
archived
```

# Recommended Indexes (as originally designed, never applied)

```sql
create index idx_memories_user_id on memories(user_id);
create index idx_goals_user_id on goals(user_id);
create index idx_projects_user_id on projects(user_id);
create index idx_tasks_user_id on tasks(user_id);
create index idx_tasks_project_id on tasks(project_id);
create index idx_tasks_plan_id on tasks(plan_id);
create index idx_plans_user_id on plans(user_id);
create index idx_decisions_user_id on decisions(user_id);
create index idx_approvals_user_id on approvals(user_id);
create index idx_executions_user_id on executions(user_id);
create index idx_reviews_user_id on reviews(user_id);
```

None of these apply — there's no `user_id` field on any Firestore document (single-user app, ownership enforced by Firestore Security Rules, not a foreign key). Firestore auto-indexes every field singly; composite indexes (for queries combining multiple `where`/`orderBy` clauses) are defined separately in a `firestore.indexes.json` file, which does not currently exist in this project. Every query in the actual store modules (`getTasks`, `getGoals`, `getProjects`, `getDecisions`) uses a single `orderBy("createdAt", "desc")` with no additional filter, which Firestore can serve without a composite index — this hasn't been a real constraint yet.

Future indexes should be added based on query patterns.

# Notes on Polymorphic Relationships

The first schema uses polymorphic junction tables for memories and reviews.

This avoids creating many separate tables too early.

If these relationships become performance-sensitive or semantically complex, they may later be split into explicit tables such as:

```text
memory_goals
memory_projects
review_goals
review_projects
```

# Notes on Status Fields

Statuses are currently represented as text values.

This keeps the V1 schema flexible.

Future implementation may move statuses to:

- database enums
- application-level constants
- lookup tables

The decision should be made during implementation.

# Outcome

This schema defined the first concrete database shape proposed for North Vector V1. It was never implemented against PostgreSQL — no real migrations were run before the Postgres/Drizzle layer was deleted.

The actual first concrete database shape is documented above in "Actual Firestore Shape (2026-07-03)," covering `tasks`, `goals`, `projects`, `decisions`, `plaid_items`, and `daily-runs`. `memories`, `plans`, `approvals`, `executions`, and `reviews` remain undesigned for any database, past or present. See `10-Implementation/ADRs/ADR-0101-Use-Firestore-as-the-Primary-Database.md`.
