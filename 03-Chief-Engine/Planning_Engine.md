# Planning Engine v1.0

## Purpose

This document defines how North Vector converts goals, priorities, deadlines, constraints, risks, and available resources into executable plans.

The Planning Engine exists to bridge the gap between intention and action.

A plan is only useful if it is realistic, sequenced, time-aware, and connected to meaningful goals.

## Core Principle

Planning is not the creation of an ideal schedule.

Planning is the design of a workable path under real constraints.

## Planning Inputs

The engine should consider:
- active goals
- current priorities
- deadlines
- events
- commitments
- available time
- available energy
- task dependencies
- risk status
- behavioral patterns
- travel and transition time
- recovery needs
- relationship obligations
- financial constraints

## Planning Horizons

### Immediate Planning

Time Horizon:
Minutes to hours

Purpose:
Determine the next action.

### Daily Planning

Time Horizon:
One day

Purpose:
Allocate time and attention across the day.

### Weekly Planning

Time Horizon:
Seven days

Purpose:
Balance major outcomes, commitments, and recovery.

### Monthly Planning

Time Horizon:
Several weeks

Purpose:
Manage milestones, exams, applications, and project progress.

### Semester Planning

Time Horizon:
Several months

Purpose:
Coordinate courses, work, clinical experience, projects, and health.

### Strategic Planning

Time Horizon:
One year or more

Purpose:
Translate major goals into phases and milestones.

## Planning Process

### Step 1: Define the Outcome

State what must be achieved.

Poor example:
`Study chemistry.`

Better example:
`Complete limiting reagents review and score at least 85% on a practice set.`

### Step 2: Identify the Deadline

Determine:
- hard deadline
- preferred completion date
- latest safe start date

### Step 3: Break the Outcome into Tasks

Tasks should be:
- concrete
- observable
- appropriately sized
- ordered
- linked to the parent goal

### Step 4: Identify Dependencies

Questions:
- What must happen first?
- What information or resource is required?
- Is another person involved?
- Is access or approval needed?

### Step 5: Estimate Effort

Estimate:
- time
- cognitive demand
- emotional demand
- physical demand
- uncertainty

Estimates should use historical data when available.

### Step 6: Add Buffer

Plans should include buffer for:
- underestimation
- delays
- fatigue
- travel
- interruptions
- unexpected work

High-risk work should receive more buffer.

### Step 7: Match Tasks to Energy

Examples:
- difficult studying during high-focus periods
- administrative tasks during lower-energy periods
- recovery after demanding events

### Step 8: Schedule Fixed Commitments

Place:
- classes
- work shifts
- appointments
- meetings
- travel

before flexible tasks.

### Step 9: Allocate Priority Work

Schedule high-priority work before optional activities.

### Step 10: Protect Health

Plans should preserve:
- adequate sleep
- meals
- exercise
- recovery
- basic personal care

### Step 11: Detect Conflicts

Check for:
- time overlap
- unrealistic density
- missing travel time
- competing deadlines
- insufficient preparation time
- energy overload

### Step 12: Produce the Plan

The plan should include:
- tasks
- order
- timing
- expected duration
- dependencies
- success criteria
- fallback options

## Task Sizing

Tasks should be small enough to begin and large enough to matter.

Bad task:
`Work on North Vector.`

Better task:
`Draft the database schema for goal records.`

Very small tasks may be grouped.

Very large tasks should be decomposed.

## Minimum Viable Plan

When uncertainty is high, Chief should create the smallest useful plan.

Example:
- identify the first action
- complete a 20-minute test block
- evaluate what remains

Action can generate information faster than extended planning.

## Daily Plan Structure

A daily plan should contain:
- primary outcome
- up to two secondary outcomes
- fixed commitments
- maintenance tasks
- protected recovery
- contingency space

The day should not be filled to 100% capacity.

## Weekly Plan Structure

A weekly plan should contain:
- three to five major outcomes
- fixed events
- preparation blocks
- project milestones
- relationship commitments
- health minimums
- catch-up space

## Planning and Behavioral Memory

Plans should account for known patterns.

Examples:

Pattern:
Delayed starts on vague work.

Adjustment:
Define the first action explicitly.

Pattern:
Optimistic scheduling.

Adjustment:
Increase estimated duration and preserve buffer.

Pattern:
New idea diversion.

Adjustment:
Keep new ideas in an intake list rather than changing the active plan immediately.

## Planning and Risk

High-risk goals should receive:
- earlier starts
- more checkpoints
- larger buffers
- clearer fallback plans
- more frequent review

## Planning and Opportunity

Plans should preserve room for valuable opportunities.

The engine should avoid scheduling so tightly that every unexpected opportunity becomes impossible.

## Plan Flexibility

Tasks should be classified as:
- fixed
- semi-flexible
- flexible

This supports dynamic rescheduling.

## Dynamic Replanning

The plan should update when:
- a task runs long
- an event changes
- energy is lower than expected
- a new risk appears
- an opportunity emerges
- a priority changes
- work is completed early

Replanning should preserve the objective rather than blindly preserving the original schedule.

## Rollover Rules

Unfinished tasks should not simply move forward forever.

When a task rolls over, Chief should ask:
- Is it still important?
- Was the estimate wrong?
- Is the task too vague?
- Is there a blocker?
- Should it be decomposed, delegated, deferred, or removed?

Repeated rollover is a diagnostic signal.

## Fallback Planning

Important plans should include fallback options.

Example:

Preferred:
Attend office hours.

Fallback:
Email the teaching assistant and complete an additional practice set.

Fallbacks preserve momentum.

## Plan Output Formats

Possible outputs:
- next-action recommendation
- time-blocked schedule
- checklist
- milestone roadmap
- weekly briefing
- recovery plan
- contingency plan

The format should match the task.

## Example Daily Plan

Primary Outcome:
Complete chemistry review target.

Fixed Commitments:
- Biology lecture, 9:00–9:50 AM
- Chemistry lecture, 10:00–10:50 AM

Plan:
- 11:30–12:15: limiting reagents review
- 1:00–1:30: lunch
- 2:00–2:45: Biology notes
- 4:30–5:00: email professor
- 7:00 onward: flexible personal time

Buffer:
One unscheduled hour.

## Planning Failure Modes

### Idealized Scheduling

Plan assumes perfect energy and no interruption.

### Overloading

Too many tasks are assigned.

### Underestimating

Durations are too optimistic.

### Missing Dependencies

Tasks are scheduled before required inputs exist.

### Health Neglect

Sleep and recovery are treated as expendable.

### Planning as Avoidance

The plan becomes more detailed while execution remains absent.

### Static Planning

The schedule does not adapt when reality changes.

## Phase 1 Implementation

Phase 1 should support:
- task decomposition
- deadlines and dependencies
- duration estimates
- fixed and flexible tasks
- daily and weekly planning
- buffers
- manual energy ratings
- dynamic rescheduling
- rollover review
- links to goals, events, and risks

Advanced optimization algorithms should come later.

## Success Criteria

The Planning Engine succeeds if Chief can reliably answer:
- What is the next action?
- What should happen today?
- What must happen this week?
- Is the plan realistic?
- What could derail it?
- What can move if circumstances change?
- How does this plan advance the larger goal?

## Final Principle

A good plan does not describe an ideal life.

A good plan survives contact with the real one.
