# Review and Reflection View Design v1.0

## Purpose

This document defines how North Vector should present daily reflections, weekly reviews, monthly strategy reviews, decision reviews, lessons, patterns, and next-step adjustments.

The Review and Reflection View exists to convert experience into learning without turning reflection into rumination, self-criticism, or administrative burden.

## Core Principle

Reflection should produce understanding.

Review should produce change.

The interface should help Nishad see what happened, why it happened, what should be repeated, what should be changed, and what action follows.

## Primary Objectives

The view should help Nishad answer:
- What happened?
- What worked?
- What failed?
- Why?
- What patterns are emerging?
- What should change?
- What should happen next?

## Review Types

The interface should support:
- Daily Reflection
- Weekly Review
- Monthly Strategy Review
- Semester Review
- Annual Review
- Decision Review
- Event Reflection
- Project Retrospective

## Default View

The default Review and Reflection View should show:
- reviews due
- recent reviews
- unresolved lessons
- recurring patterns
- recommended changes
- next review date

Completed and archived reviews should remain searchable.

## Review Card

Each review card should include:
- review type
- date or period
- executive summary
- status
- major win
- major miss
- top lesson
- next action

Possible quick actions:
- Open Review
- Continue
- Mark Complete
- Add Reflection
- Apply Recommendation
- Archive

## Daily Reflection View

The Daily Reflection should remain lightweight.

Suggested prompts:
- What mattered today?
- What was completed?
- What was missed?
- What created friction?
- What worked unusually well?
- What should carry into tomorrow?

The interface should not require a long journal entry.

## Weekly Review View

The Weekly Review should include:
- executive summary
- major wins
- major misses
- goal progress
- priority review
- accountability review
- risk review
- opportunity review
- health review
- relationship review
- behavioral patterns
- lessons
- next-week priorities

The interface should clearly separate observation from decision.

## Monthly Strategy Review View

The Monthly Strategy Review should emphasize:
- mission alignment
- goal portfolio
- resource allocation
- academic strategy
- career strategy
- financial strategy
- health strategy
- relationship strategy
- project portfolio
- recurring risks
- emerging opportunities
- strategic decisions

This view should support explicit continue, change, pause, or stop decisions.

## Decision Review View

The Decision Review should show:
- original decision
- options considered
- information available at the time
- assumptions
- recommendation
- action taken
- actual outcome
- decision quality
- process errors
- lessons

The interface should separate:
- good decision, good outcome
- good decision, bad outcome
- weak decision, good outcome
- weak decision, bad outcome

## Reflection Entry

Each reflection entry should include:
- date
- domain
- event or objective
- observation
- interpretation
- lesson
- confidence
- related goals
- related memories
- next action

## Observation vs Interpretation

The interface should visually distinguish:

### Observation

What happened.

Example:
`Two planned chemistry review blocks were missed.`

### Interpretation

What it may mean.

Example:
`The study plan may have been too optimistic.`

### Lesson

What should change.

Example:
`Use shorter blocks and start one day earlier.`

This distinction reduces false certainty.

## Success Review

The interface should analyze success, not only failure.

Suggested prompts:
- What worked?
- Why did it work?
- Was it repeatable?
- Which conditions helped?
- Should this become a success pattern?

## Failure Review

The interface should analyze failure without moralizing.

Suggested prompts:
- What happened?
- What was underestimated?
- What was avoided?
- Which system failed?
- What intervention would have helped?

## Pattern Detection

The view should surface candidate patterns from repeated reviews.

Example:
`Ambiguous tasks were delayed in three of the last four weeks.`

Possible actions:
- Confirm Pattern
- Add Evidence
- Reject Pattern
- Keep Watching

## Lesson Capture

Lessons should be concise and actionable.

Good:
`Front-load high-risk science work.`

Weak:
`Try harder next time.`

Each lesson should connect to:
- a changed rule
- a planning adjustment
- a risk rule
- a behavioral memory
- a next action

## Recommendation Application

The interface should let Nishad apply a lesson directly.

Examples:
- update study buffer
- change reminder timing
- revise a goal
- create a new planning rule
- archive a project

Reflection should not end as static text.

## Review Timeline

A timeline may show:
- reviews
- major events
- decisions
- lessons
- pattern confirmations

This helps identify change over time.

## Comparison View

Allow comparisons between periods.

Examples:
- this week vs last week
- this month vs previous month
- current semester vs previous semester

Compare:
- goal progress
- completion reliability
- risk count
- opportunity capture
- health protection
- recurring patterns

Comparisons should remain diagnostic, not competitive.

## Strategic Alignment Indicator

Each major review may include a simple alignment assessment.

Suggested labels:
- Strongly Aligned
- Mostly Aligned
- Mixed
- Drifting
- Misaligned

The label should be explained in plain language.

## Review Prompts

Prompts should adapt to context.

Example after exam:
- Was preparation early enough?
- Which topics remained weak?
- Did the plan match the actual exam?

Example after project milestone:
- What blocked progress?
- Was scope controlled?
- Did planning become avoidance?

## Review Completion

Completing a review should produce:
- concise summary
- confirmed lessons
- updated memories
- changed plans or rules
- next actions

The system should show what changed because of the review.

## Reminder Design

Review reminders should be low-pressure and specific.

Example:
`Your weekly review is due. It should take about 10 minutes.`

Missed reviews should not become a guilt loop.

## Review Depth Options

Offer:
- Quick Review
- Standard Review
- Deep Review

This prevents the process from becoming burdensome.

## Search and Filters

Allow filtering by:
- review type
- date
- domain
- goal
- project
- person
- lesson
- pattern
- alignment status

## Visual Design

The view should feel:
- calm
- thoughtful
- structured
- honest
- nonjudgmental

Avoid:
- red failure dashboards
- excessive scoring
- motivational clichés
- forced journaling aesthetics

## Mobile View

On phone, prioritize:
- review due
- quick prompts
- top lesson
- next action
- voice reflection

## Wearable View

For glasses or watch, support only:
- quick reflection capture
- one question at a time
- handoff to phone or MacBook for full review

## Accessibility

The view should support:
- keyboard navigation
- screen readers
- scalable text
- voice input
- one-question-at-a-time mode
- clear section labels

## Empty States

No reviews due:
`No reviews are currently due.`

No confirmed patterns:
`No behavioral patterns have enough evidence yet.`

No unresolved lessons:
`All recent lessons have been applied or archived.`

## Error States

If data is incomplete:
`This review may be missing recent calendar or task data.`

The interface should show which sources failed.

## Failure Modes

### Rumination

Reflection repeats the problem without producing change.

### Shame Design

The interface frames mistakes as character defects.

### Excessive Review

The process takes more time than the lesson is worth.

### Weak Lessons

Insights remain vague and unactionable.

### No System Update

The review produces text but changes nothing.

### Outcome Bias

A good or bad result dominates process analysis.

### Score Obsession

Metrics replace understanding.

## Phase 1 Implementation

Phase 1 should include:
- daily reflection
- weekly review
- monthly strategy review
- decision review
- review cards
- structured prompts
- lesson capture
- pattern candidates
- next-action generation
- memory and plan updates

Advanced comparison analytics can come later.

## Success Criteria

The Review and Reflection View succeeds if Nishad can quickly understand:
- what happened
- why it happened
- what worked
- what failed
- what should change
- what action follows

## Final Principle

Reflection should not merely explain the past.

It should improve the next decision.
